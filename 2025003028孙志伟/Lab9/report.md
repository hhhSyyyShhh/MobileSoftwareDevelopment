# Lab9 甜品点击器应用实验报告

## 1. ViewModel 在 Android 架构中的作用简述
ViewModel 是 Android 架构组件的核心，用于管理 UI 相关数据与业务逻辑。它独立于 UI 生命周期，在屏幕旋转、配置更改时不会被销毁，能够有效保存数据状态；同时将 UI 展示与数据处理分离，降低代码耦合，让页面更简洁、逻辑更集中，是 MVVM 架构的关键组成部分。

## 2. `DessertUiState` 数据类的字段设计说明
DessertUiState 用于统一管理界面所需的所有状态数据，字段设计如下：
- `revenue`：记录当前总收入，默认值为 0；
- `dessertsSold`：记录已售出甜品数量，默认值为 0；
- `currentDessertIndex`：当前甜品在列表中的索引，默认值为 0；
- `currentDessertImageId`：当前甜品的图片资源 ID，默认使用纸杯蛋糕；
- `currentDessertPrice`：当前甜品的单价，默认值为 5。
所有数据集中管理，让 UI 只负责读取展示，不负责存储与计算。

## 3. `DessertViewModel` 的设计思路
### 状态管理
- 使用 `mutableStateOf` 包装 `DessertUiState`，使 Compose 可以自动监听状态变化；
- 使用 `private set` 限制外部修改，保证数据安全，只能通过内部方法更新状态。

### 方法设计
- `onDessertClicked()`：处理甜品点击事件，更新收益和销量，并自动切换甜品；
- `determineDessertToShow()`：根据销量判断当前应显示的甜品，纯业务逻辑，与 UI 完全分离；
- 从 Datasource 读取甜品数据，实现数据来源与业务逻辑解耦。

## 4. `MainActivity` 重构前后对比分析
### 重构前
- 状态变量直接写在 UI 中，代码混乱；
- 业务逻辑与界面代码混合在一起，难以维护；
- 屏幕旋转后数据会丢失，体验差；
- 所有逻辑集中在一个文件，结构不清晰。

### 重构后
- UI 不管理任何数据，只负责展示和触发事件；
- 所有状态和逻辑交给 ViewModel 处理；
- 旋转屏幕数据不丢失，生命周期更安全；
- 代码职责明确，可读性和可维护性大幅提升。

## 5. 重构前后代码结构的区别和感受
### 结构区别
- 重构前：单一文件结构，UI、逻辑、数据混合；
- 重构后：分层结构清晰，分为 model、data、ui、ViewModel 等模块，各司其职。

### 感受
重构后代码更加清晰、易于修改和扩展，逻辑与界面分离后，排查错误更高效，程序稳定性更强，也更符合 Android 官方开发规范，整体编写体验和项目质量都有明显提升。

## 6. 遇到的问题与解决过程
1. **viewModel() 无法使用**
   原因：未添加 ViewModel 依赖库
   解决：在 build.gradle 中添加 lifecycle-viewmodel-compose 依赖并同步。

2. **运行按钮灰色不可点击**
   原因：代码存在语法错误或资源引用错误
   解决：检查红色报错位置，修正包名、文件名与资源 ID。

3. **Android Studio 运行旧代码**
   原因：构建缓存未更新
   解决：执行 Clean Project → Rebuild Project 清理缓存。

4. **旋转屏幕数据丢失**
   原因：状态保存在 UI 中，未使用 ViewModel
   解决：将所有状态移入 ViewModel，实现生命周期安全。

5. **包名不一致导致报错**
   原因：项目包名与代码包名不匹配
   解决：统一包名并重构目录结构，确保引用正确。
