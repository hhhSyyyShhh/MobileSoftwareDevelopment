# Lab8：超级英雄列表应用
## 一、应用功能与展示信息
本项目基于 Jetpack Compose 实现纯声明式 UI 开发，是一个典型的单 Activity 列表类应用，整体遵循数据层 - UI 层 - 主题层分离的架构设计:
- 展示超级英雄名称、描述、头像的滚动列表
- 支持浅色 / 深色模式自动切换
- 统一的 Material Design 3 主题样式
- 沉浸式状态栏与顶部应用栏适配
- 实现数据与界面分离，便于后续扩展英雄内容与修改界面样式。
---

## 二、布局结构说明
本次界面采用模块化、分层嵌套的结构设计，各组件职责清晰，布局逻辑明确：
1.  **根布局**：使用 `Scaffold` 作为页面根容器，填充整个屏幕空间，适配系统状态栏与导航栏，提供稳定的页面框架。
2.  **列表布局**：使用 `LazyColumn` 实现可滚动列表，配置内边距与卡片间距，实现高效滚动渲染，仅加载屏幕可见区域卡片，提升应用运行性能。
3.  **卡片容器**：使用 `Card` 组件实现卡片样式，配置圆角、阴影与主题背景色，提升界面视觉层次，适配深浅两种主题模式。
4.  **卡片内部布局**：采用 `Row` 水平布局，将文字区域与头像区域左右排列：
    - 左侧文字区域：使用 `Column` 垂直布局，依次展示英雄名称与能力描述，标题与描述采用不同字体层级，保证信息层级清晰。
    - 右侧头像区域：固定尺寸72dp，添加8dp圆角，按比例裁剪显示，保证图片不变形。

整体布局遵循 Compose 最佳实践，组件复用性强，结构易于维护。

## 三、关键技术实现
### 1. 数据模型与数据源实现
本次实验采用数据与界面分离的设计思想，将数据定义与界面渲染完全解耦：
- 定义 `Hero` 数据类，使用资源 ID 存储英雄名称、描述和头像信息，避免硬编码字符串和图片，便于后续资源替换与国际化扩展。
- 使用单例对象 `HeroesRepository` 作为数据源，统一管理所有英雄数据，提供静态列表供界面直接调用，结构简单清晰，便于维护和扩展。

### 2. 可滚动列表实现
通过 `LazyColumn` 实现高性能可滚动列表：
- 配置 `contentPadding` 设置列表整体内边距，保证卡片与屏幕边缘有16dp留白。
- 配置 `verticalArrangement.spacedBy(8.dp)` 设置卡片之间的垂直间距，符合设计规范。
- 使用 `items()` 方法渲染列表项，仅加载屏幕可见区域的卡片，提升列表滚动性能。
- 结合 `Scaffold` 的 `innerPadding`，避免列表内容被顶部应用栏遮挡。

### 3. 界面样式与边距控制
通过 `Modifier` 修饰符精准控制界面样式，严格匹配设计规范：
- 卡片使用 `MaterialTheme.shapes.medium` 圆角形状，实现16dp圆角效果。
- 卡片背景色使用 `secondaryContainer` 主题色，自动适配深浅两种模式。
- 卡片添加 `elevation` 阴影效果，提升视觉层次。
- 头像添加 `RoundedCornerShape(8.dp)` 圆角，与设计稿保持一致。
- 文字区域使用 `weight(1f)` 占满剩余空间，保证文字与头像的布局比例固定。

### 4. 自定义主题实现
本次实验实现了完整的 Material 3 自定义主题，包含：
- **颜色配置**：分别配置浅色 / 深色模式下的背景色、文字色、主色调;支持系统深色模式自动切换
- **字体配置**：导入自定义 `Cabin` 字体，全局文字风格统一，提升应用质感
- **形状配置**：统一圆角大小，卡片、按钮样式一致,符合现代 UI 设计规范

### 5. 状态栏适配
- 隐藏状态栏背景色块，实现内容延伸至状态栏下方的视觉效果。
- 深浅模式下状态栏图标颜色自动适配，整体视觉统一，无割裂感。

## 四、遇到的问题与解决过程
1.  **资源文件命名不规范导致编译报错**：字体文件与图片文件名包含大写字母、特殊符号，导致编译报错。解决：将文件名修改为全小写+下划线格式，符合Android资源命名规范。
2.  **列表项文字被裁剪问题**：卡片设置了固定高度，导致多行描述文字被裁剪。解决：移除固定高度，让卡片高度自适应内容。
3.  **包声明不匹配问题**：文件路径与包声明不一致，导致导入错误、全线标红。解决：统一所有文件的包声明为项目根包名，同步项目路径。
4.  **顶部栏遮挡列表内容问题**：未处理 `Scaffold` 的 `innerPadding`，导致列表内容被顶部应用栏遮挡。解决：在 `LazyColumn` 中传入 `innerPadding`，适配顶部栏高度。
