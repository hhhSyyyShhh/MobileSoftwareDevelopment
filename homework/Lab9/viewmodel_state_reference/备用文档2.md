## 1. 前言：为什么要学 ViewModel？

### 1.1 一个生活中的故事

想象你在玩一个手机游戏，已经玩到第 8 关，得了 160 分。突然你想把手机从竖屏转成横屏——

**屏幕一旋转，游戏回到第 1 关，分数归零。**

你是不是想摔手机？

这就是在学 ViewModel 之前，Android 应用会遇到的问题。屏幕旋转在 Android 系统中被称为"**配置更改（Configuration Change）**"——当旋转发生时，Android 系统会**销毁当前的 Activity 然后重新创建它**。

你可以把它想象成：
- 你正在一张白纸上写字（屏幕上显示游戏状态）
- 旋转屏幕 = 老师把纸收走，给你一张新纸
- 你之前写的东西全没了！

### 1.2 之前学过的解决方案及其局限

在这门课的前面部分，你学过 `rememberSaveable`，它可以在旋转时保存一些简单的数据。这就像你在纸上贴了一张便利贴——纸上内容丢了，便利贴还在。

但问题是：
- 如果应用越来越复杂，把所有的数据和逻辑都写在可组合函数（Composable）里面，代码会变得像一团乱麻
- 你无法方便地测试这些逻辑
- 数据和界面混在一起，修改一处可能破坏另一处

### 1.3 本教程的解决方案

我们会学习三个"神器"：

| 神器 | 一句话解释 | 类比 |
|------|-----------|------|
| **ViewModel** | 一个专门帮你保管数据的"管家" | 你把重要文件交给管家保管，房子塌了重建，管家还在，文件不丢 |
| **UI State（界面状态）** | 用数据描述界面"长什么样" | 给界面拍一张"数据X光片"，所有显示内容都在里面 |
| **UDF（单向数据流）** | 数据只往一个方向流动 | 像水流一样，只能从上游（ViewModel）流向下游（UI）；事件像气泡，从下游（UI）冒到上游（ViewModel） |

### 1.4 本教程的前提条件

- 你会 Kotlin 基础：函数怎么写、lambda 表达式（`{ it -> doSomething(it) }`）是什么意思
- 你会用 Compose 搭基本的布局：`Row`、`Column`、`Text`、`Button`
- 你知道 Material Design 大概是什么
- 你安装了 Android Studio

### 1.5 本教程的目标

完成之后，你会：
- ✨ 理解 Android 应用架构的基本思想
- ✨ 会创建和使用 `ViewModel`
- ✨ 会定义和更新 UI State
- ✨ 会实现 UDF 模式
- ✨ 搞定设备旋转不丢数据的问题

## 2. 应用概览：我们要做什么？

### 2.1 Unscramble 猜词游戏规则

我们要做一个叫 **Unscramble** 的猜词游戏：

```
┌──────────────────────────────┐
│         Unscramble           │
│  ┌────────────────────────┐  │
│  │    Word: 1/10          │  │
│  │                        │  │
│  │    ZYCR A  ← 乱序词    │  │
│  │                        │  │
│  │  Enter the scrambled   │  │
│  │  word, and tap Submit  │  │
│  │  ┌──────────────────┐  │  │
│  │  │  your guess...   │  │  │
│  │  └──────────────────┘  │  │
│  └────────────────────────┘  │
│  [  Submit  ]  [  Skip  ]   │
│        Score: 80             │
└──────────────────────────────┘
```

**规则很简单：**

1. 显示一个乱序词（比如 "CRYZA"），实际是 "CRAZY" 打乱的
2. 玩家输入猜到的原词
3. 猜对 → +20 分 → 下一个词
4. 猜错 → 显示 "Wrong Guess!" → 可以重新猜或跳过
5. 可以点 Skip 跳过当前词（不扣分）
6. 一局共 10 个词
7. 10 个词结束 → 弹窗显示总分，可以再玩一局或退出

## 3. 起始应用概览：理解已有代码

### 3.1 起始代码有什么？

运行起始代码你会看到：界面布局已经有了，但**乱序词固定显示 "scrambleun"，点任何按钮都没反应**。这就是我们要修复的问题。

先来看看已有的代码结构：

```
unscramble/
├── data/
│   └── WordsData.kt          ← 词库和游戏常量
├── ui/
│   └── GameScreen.kt         ← 所有界面代码（我们要改的主要文件）
├── MainActivity.kt           ← Activity 入口
└── res/
    └── values/
        └── strings.xml        ← 字符串资源
```

### 3.2 WordsData.kt —— 词库文件

```kotlin
package com.example.android.unscramble.data

const val MAX_NO_OF_WORDS = 10       // 每局最多10个词
const val SCORE_INCREASE = 20        // 每猜对一个加20分

val allWords: Set<String> = setOf(
    "animal", "auto", "anecdote", "alphabet", "all",
    "awesome", "arise", "balloon", "basket", "bench",
    // ... 还有很多词 ...
    "zoology", "zone", "zeal"
)
```

**逐行解释：**

| 代码 | 含义 | 生活类比 |
|------|------|---------|
| `const val MAX_NO_OF_WORDS = 10` | 每局游戏最多10个词 | 一副扑克牌规定最多发10张 |
| `const val SCORE_INCREASE = 20` | 猜对一个词加20分 | 考试一道题20分 |
| `val allWords: Set<String> = setOf(...)` | 所有可选词汇的集合 | 一个装满单词卡片的盒子 |
| `Set<String>` | 一个不允许重复元素的集合 | 一盒每张都不一样的单词卡片 |

> **`Set` 和 `List` 的区别**：Set 里的元素不能重复，就像身份证号不能重复；List 可以重复，像购物清单可以写两次"苹果"。

### 3.3 GameScreen.kt —— 界面文件（起始版）

这是核心文件，包含了 4 个可组合函数。我们逐一分析。

#### 3.3.1 GameStatus — 显示分数

```kotlin
@Composable
fun GameStatus(score: Int, modifier: Modifier = Modifier) {
    Card(modifier = modifier) {
        Text(
            text = stringResource(R.string.score, score),
            style = typography.headlineMedium,
            modifier = Modifier.padding(8.dp)
        )
    }
}
```

**逐行解释：**

- `@Composable`：告诉 Kotlin 这是一个"可组合函数"——一个可以画到屏幕上的 UI 组件
- `score: Int`：接受一个整数作为分数
- `modifier: Modifier = Modifier`：Modifier 相当于"装饰参数"，控制组件的大小、位置、边距等。有默认值所以可以不传
- `Card(...)`：Material 3 的卡片组件，像一个带阴影的白色方框
- `stringResource(R.string.score, score)`：从 strings.xml 读取字符串模板，把 `score` 的值填入。例如 score=80 时显示 "Score: 80"
- `typography.headlineMedium`：使用 Material 3 主题里预设的"中号大标题"字体样式
- `.padding(8.dp)`：在文字周围留 8dp 的空白。dp 是 Android 的密度无关像素单位

**这个函数画出来的效果：**

```
┌──────────────┐
│  Score: 80   │
└──────────────┘
```

#### 3.3.2 GameLayout — 主要游戏区域（起始版）

这是最长的函数，包含乱序词、说明文字和输入框。

```kotlin
@Composable
fun GameLayout(modifier: Modifier = Modifier) {
    val mediumPadding = dimensionResource(R.dimen.padding_medium)

    Card(
        modifier = modifier,
        elevation = CardDefaults.cardElevation(defaultElevation = 5.dp)
    ) {
        Column(
            verticalArrangement = Arrangement.spacedBy(mediumPadding),
            horizontalAlignment = Alignment.CenterHorizontally,
            modifier = Modifier.padding(mediumPadding)
        ) {
            // ① 右上角的单词计数
            Text(
                modifier = Modifier
                    .clip(shapes.medium)              // 裁剪为圆角
                    .background(colorScheme.surfaceTint) // 背景色
                    .padding(horizontal = 10.dp, vertical = 4.dp)
                    .align(alignment = Alignment.End),   // 靠右对齐
                text = stringResource(R.string.word_count, 0),
                style = typography.titleMedium,
                color = colorScheme.onPrimary
            )

            // ② 乱序词（硬编码 bug）
            Text(
                text = "scrambleun",         // ← 问题所在！写死了
                style = typography.displayMedium
            )

            // ③ 说明文字
            Text(
                text = stringResource(R.string.instructions),
                textAlign = TextAlign.Center,
                style = typography.titleMedium
            )

            // ④ 输入框
            OutlinedTextField(
                value = "",                  // ← 输入内容永远为空
                singleLine = true,
                shape = shapes.large,
                modifier = Modifier.fillMaxWidth(),
                colors = TextFieldDefaults.textFieldColors(
                    containerColor = colorScheme.surface
                ),
                onValueChange = { },         // ← 空回调，输入无反应
                label = { Text(stringResource(R.string.enter_your_word)) },
                isError = false,
                keyboardOptions = KeyboardOptions.Default.copy(
                    imeAction = ImeAction.Done
                ),
                keyboardActions = KeyboardActions(
                    onDone = { }             // ← 空回调，键盘Done无反应
                )
            )
        }
    }
}
```

**逐行详细解释：**

**Card 部分：**

| 参数 | 含义 | 为什么这样写 |
|------|------|-------------|
| `modifier = modifier` | 让外部可以控制这个 Card 的大小和位置 | 灵活性——调用者可以决定卡片占多宽 |
| `elevation = CardDefaults.cardElevation(defaultElevation = 5.dp)` | 给卡片加 5dp 的阴影高度 | 让卡片看起来"浮"在背景之上 |

**Column 部分：**

`Column` 是一个"纵向排列容器"——把子元素从上到下一个接一个地排列。

| 参数 | 含义 |
|------|------|
| `verticalArrangement = Arrangement.spacedBy(mediumPadding)` | 子元素之间间隔 mediumPadding（中等间距） |
| `horizontalAlignment = Alignment.CenterHorizontally` | 子元素在水平方向居中 |

**① 单词计数 Text：**

```kotlin
Text(
    modifier = Modifier
        .clip(shapes.medium)               // 第1步：裁剪成中等圆角形状
        .background(colorScheme.surfaceTint) // 第2步：涂上主题色背景
        .padding(horizontal = 10.dp, vertical = 4.dp) // 第3步：加内边距
        .align(alignment = Alignment.End), // 第4步：在 Column 里靠右
    ...
)
```

> **Modifier 链的顺序很重要！**它们是"从外到内"依次应用的。先 clip 后 background 意味着"先裁剪形状，再在这个形状里涂色"。如果反过来，背景色会涂到裁剪区域之外。

**② 乱序词 Text：**

```kotlin
Text(
    text = "scrambleun",       // ← 这是起始代码的 bug
    ...
)
```

这个值是**硬编码（hardcoded）**的——每次运行都显示一样的文字。我们需要把它改成从 ViewModel 动态获取。

**③ 说明文字 Text：**

没什么特殊，居中显示一些提示文字，比如 "Enter the scrambled word, and tap Submit"。

**④ OutlinedTextField 输入框：**

这是 Material 3 的"线框风格"文本输入框。和普通 `TextField` 对比：

| 类型 | 外观 | 适用场景 |
|------|------|---------|
| `TextField`（FilledTextField） | 有填充背景色 | 单个重要输入框 |
| `OutlinedTextField` | 只有边框线 | 表单中多个输入框 |

起始版的问题：
- `value = ""`：输入框内容永远为空字符串
- `onValueChange = { }`：当用户输入时，回调是空的——什么都不做
- `onDone = { }`：键盘上按 Done 键，回调也是空的

这就是为什么起始版中你输入不了任何东西——输入事件被丢弃了。

**keyboardOptions 和 keyboardActions：**

```kotlin
keyboardOptions = KeyboardOptions.Default.copy(
    imeAction = ImeAction.Done    // 键盘右下角显示 "Done" 按钮
),
keyboardActions = KeyboardActions(
    onDone = { }                  // 按 Done 后的操作（目前为空）
)
```

`imeAction = ImeAction.Done` 把键盘右下角的按钮文字改成 "Done"（完成）。IME = Input Method Editor（输入法编辑器）。

#### 3.3.3 GameScreen — 游戏页面主体（起始版）

```kotlin
@Composable
fun GameScreen() {
    val mediumPadding = dimensionResource(R.dimen.padding_medium)

    Column(
        modifier = Modifier
            .verticalScroll(rememberScrollState())  // 内容太长时可以滚动
            .padding(mediumPadding),
        verticalArrangement = Arrangement.Center,
        horizontalAlignment = Alignment.CenterHorizontally
    ) {
        // 标题
        Text(
            text = stringResource(R.string.app_name),
            style = typography.titleLarge
        )

        // 游戏主区域
        GameLayout(
            modifier = Modifier
                .fillMaxWidth()
                .wrapContentHeight()
                .padding(mediumPadding)
        )

        // Submit 和 Skip 按钮
        Column(
            modifier = Modifier
                .fillMaxWidth()
                .padding(mediumPadding),
            verticalArrangement = Arrangement.spacedBy(mediumPadding),
            horizontalAlignment = Alignment.CenterHorizontally
        ) {
            Button(
                modifier = Modifier.fillMaxWidth(),
                onClick = { }              // ← 空回调！点 Submit 没反应
            ) {
                Text(
                    text = stringResource(R.string.submit),
                    fontSize = 16.sp
                )
            }

            OutlinedButton(
                onClick = { },             // ← 空回调！点 Skip 没反应
                modifier = Modifier.fillMaxWidth()
            ) {
                Text(
                    text = stringResource(R.string.skip),
                    fontSize = 16.sp
                )
            }
        }

        // 分数显示
        GameStatus(
            score = 0,                     // ← 硬编码为 0
            modifier = Modifier.padding(20.dp)
        )
    }
}
```

**逐行解释关键点：**

- `rememberScrollState()`：让 Column 可滚动。手机屏幕小，内容多时需要滚动
- `Arrangement.Center`：在纵向方向上居中排列（顶部和底部各有等量空白）
- `GameLayout(...)`：调用了我们上面分析的函数
- `Button` vs `OutlinedButton`：
  - `Button` = 实心按钮（有背景填充色），用于主要操作
  - `OutlinedButton` = 线框按钮（只有边框），用于次要操作
- `onClick = { }`：两个按钮的回调都是空的——所以点击无反应

#### 3.3.4 FinalScoreDialog — 游戏结束对话框（起始版）

```kotlin
@Composable
private fun FinalScoreDialog(
    score: Int,
    onPlayAgain: () -> Unit,
    modifier: Modifier = Modifier
) {
    val activity = (LocalContext.current as Activity)

    AlertDialog(
        onDismissRequest = { },
        title = { Text(text = stringResource(R.string.congratulations)) },
        text = { Text(text = stringResource(R.string.you_scored, score)) },
        modifier = modifier,
        dismissButton = {
            TextButton(onClick = { activity.finish() }) {
                Text(text = stringResource(R.string.exit))
            }
        },
        confirmButton = {
            TextButton(onClick = onPlayAgain) {
                Text(text = stringResource(R.string.play_again))
            }
        }
    )
}
```

**逐行解释：**

- `private fun`：这个函数是私有的，只在当前文件内可见
- `onPlayAgain: () -> Unit`：一个 lambda 回调，"Unit" 表示没有返回值。当用户点 "Play Again" 时调用
- `val activity = (LocalContext.current as Activity)`：获取当前的 Activity 对象，用于点击 Exit 时关闭应用
- `AlertDialog`：Material 3 的"提醒对话框"——一个小弹窗覆盖在界面上方

**AlertDialog 的结构：**

```
┌──────────────────────────────┐
│      Congratulations!        │  ← title（标题）
│                              │
│  You scored: 160             │  ← text（正文内容）
│                              │
│  ────────────────────────    │  ← 可选的 divider（分隔线）
│                              │
│  [ Exit ]    [ Play Again ]  │  ← dismiss/confirm buttons
└──────────────────────────────┘
```

- `dismissButton`：通常放"取消"或"退出"这类次要操作
- `confirmButton`：通常放"确认"或"再来一局"这类主要操作
- `onDismissRequest = { }`：当用户点对话框外部或按返回键时触发。起始版是空的，意味着用户**无法关闭对话框**——这其实是一个设计选择：游戏结束了要么再玩一局要么退出

### 3.4 起始代码问题总结

| 问题 | 具体表现 | 根本原因 |
|------|---------|---------|
| 乱序词不变 | 永远显示 "scrambleun" | 硬编码字符串 |
| 输入无效 | 键盘输入不显示 | `onValueChange` 是空函数 |
| 按钮无反应 | Submit/Skip 不工作 | `onClick` 是空函数 |
| 分数不变 | 永远显示 0 | `score = 0` 硬编码 |
| 旋转丢数据 | 旋转屏幕后状态丢失 | 没有使用 ViewModel |

这就像一台玩具手机——外壳漂亮，里面的电路全是假的。我们的任务是把"真电路"装上。

## 4. 了解应用架构：编程界的"楼房设计图"

在动手写代码之前，先要理解"为什么要这样设计"。就像盖房子之前需要看设计图一样。

### 4.1 两大核心原则

#### 原则一：分离关注点 (Separation of Concerns)

**一句话：不同的事情交给不同的人做。**

生活类比：
```
厨房里：
  - 主厨负责炒菜（业务逻辑）
  - 切菜工负责备菜（数据处理）
  - 服务员负责上菜（界面展示）

如果让主厨又炒菜又上菜又收银，不乱才怪！
```

对应到 Android 应用：
```
应用里：
  - ViewModel 负责处理数据和逻辑（主厨）
  - Data Layer 负责存取数据（切菜工）
  - Composable UI 负责画界面（服务员）
```

#### 原则二：通过模型驱动界面 (Drive UI from Model)

**一句话：界面长什么样，由数据说了算。数据变了，界面自动跟着变。**

生活类比：
```
遥控器（UI）←→ 电视机内部状态（Model）

你不是直接用手去掰电视画面让它变色，
而是按遥控器按钮 → 改变电视机内部状态 → 电视画面自动改变
```

这就是 Compose 的核心思想：**你只管改数据，界面自动更新。**

### 4.2 推荐的应用架构层次

Android 官方推荐的应用架构至少分为两层：

```
┌─────────────────────────────┐
│      界面层 (UI Layer)      │  ← 用户能看到和互动的
│  ┌───────────────────────┐  │
│  │  UI Elements          │  │  ← Composable 函数画的界面
│  │  (Compose 组件)       │  │
│  ├───────────────────────┤  │
│  │  State Holders        │  │  ← ViewModel 保管数据
│  │  (ViewModel)          │  │
│  └───────────────────────┘  │
├─────────────────────────────┤
│     数据层 (Data Layer)      │  ← 存取数据的
│  Repository → DataSource    │
└─────────────────────────────┘
     ↑ 可选
┌─────────────────────────────┐
│   网域层 (Domain Layer)      │  ← 封装可复用的业务逻辑
│   UseCase                   │     (本项目用不到)
└─────────────────────────────┘
```

**生活类比：**

| 层 | 职责 | 餐厅类比 |
|----|------|---------|
| 界面层 | 画界面 + 管理界面状态 | 服务员：摆盘、端盘、记录点单 |
| 数据层 | 存取数据（数据库、网络） | 冷库：存储食材 |
| 网域层 | 可复用的业务逻辑 | 标准菜谱：不管哪个厨师做，都按菜谱来 |

### 4.3 ViewModel 是什么？

**ViewModel = 界面的"数据管家"**

```kotlin
// ViewModel 就是一个普通的 Kotlin 类，只是继承自 ViewModel
class GameViewModel : ViewModel() {
    // 这里放游戏需要的数据和处理逻辑
}
```

**ViewModel 的三大超能力：**

1. **存数据**：可以持有界面需要的所有数据
2. **活得更久**：Activity 被销毁重建（如旋转屏幕）时，ViewModel 不会被销毁
3. **与界面解耦**：ViewModel 不持有任何 UI 引用，不会造成内存泄漏

**生命周期的秘密：**

```
Activity 生命周期：  创建 → ... → 旋转屏幕 → 销毁 → 重建 → ...
                            ↓
ViewModel 生命周期：  创建 → 一直存活 → 直到 Activity 真正结束
```

就像：
- Activity 是一只蝴蝶，生命周期很短
- ViewModel 是乌龟，活很久——蝴蝶死了又重生，乌龟还在那里

> **⚠️ 重要限制**：ViewModel 只在"Activity 被系统主动销毁重建"时保留数据。如果用户主动关闭应用（进程被杀死），ViewModel 也会跟着死。它不能代替数据库或文件存储！

### 4.4 界面状态（UI State）是什么？

**UI State = 一张"数据快照"，描述界面此时此刻应该显示什么**

```kotlin
// 这就是一个 UI State 的定义
data class GameUiState(
    val currentScrambledWord: String = "",   // 当前乱序词
    val score: Int = 0,                       // 当前分数
    val currentWordCount: Int = 1,            // 当前第几个词
    val isGuessedWordWrong: Boolean = false,  // 是否猜错了
    val isGameOver: Boolean = false           // 游戏是否结束
)
```

**这个数据类包含了"界面显示需要的全部信息"。**

生活类比：
```
去餐厅点菜，你给服务员一张点菜单：
┌─────────────────────┐
│ 菜名：宫保鸡丁      │
│ 辣度：微辣          │
│ 米饭：要            │
│ 已上菜：否          │
└─────────────────────┘

这张点菜单 = UI State
厨师做菜的过程 = ViewModel 的业务逻辑
服务员端上来的菜 = 界面显示
```

### 4.5 不可变性（Immutability）——为什么用 `val` 和 `data class`？

**一句话：UI State 创建后就不能直接修改，要改就创建一个新的副本。**

```kotlin
// ❌ 不好的做法：直接修改
state.score = 80     // 直接改了！别人如果也在读 score 就会出 bug

// ✅ 好的做法：创建副本
val newState = state.copy(score = 80)   // 创建一个新的，旧的不变
```

**为什么？**

生活类比：
```
你和朋友共享一份 Google 文档
- 如果每个人都能直接改原文（可变） → 你改一个字，别人看到的瞬间就变了，谁也不知道改了什么
- 如果每次修改都创建一个新版本（不可变） → 你可以看到"v1→v2→v3"的完整变化历史，不会混乱
```

**技术上：**
- `data class` 自动生成 `copy()` 方法，让你方便地创建"只改了一个字段"的副本
- `val`（不是 `var`）确保每个字段在创建后都不能被直接修改

### 5.3 步骤 3：创建 GameUiState 数据类

**做什么：** 定义游戏界面状态的"格式"。

在 `GameViewModel.kt` 中（可以在同一个文件里），添加：

```kotlin
data class GameUiState(
    val currentScrambledWord: String = ""
)
```

**逐行解释：**

- `data class`：Kotlin 的"数据类"关键字。与普通 class 的区别是，编译器会自动生成：
  - `equals()` 方法（判断两个对象内容是否相同）
  - `hashCode()` 方法
  - `toString()` 方法（打印时显示内容而非内存地址）
  - `copy()` 方法（创建副本，可以只改部分字段）

- `GameUiState`：命名惯例——描述的是"Game 的 UI 的 State"
- `val currentScrambledWord: String = ""`：
  - `val` = 不可变，创建后不能直接改
  - `currentScrambledWord` = 当前显示的乱序词
  - `String` = 类型是字符串
  - `= ""` = 默认值是空字符串

> **为什么 `currentScrambledWord` 默认是空字符串？** 因为 Kotlin 强烈推荐给属性默认值。在创建 `GameUiState()` 时如果不传参数，就使用默认值。

### 5.4 步骤 4：引入 StateFlow ——"可观察的数据流"

#### 5.4.1 什么是 StateFlow？

**StateFlow = 一个会"通知"别人的数据容器**

生活类比：
```
普通变量 = 一张贴在墙上的便签，你得主动走过去看才知道写了什么
StateFlow = 一个群聊消息，有人改了数据，"叮"一声所有人都知道新值是什么
```

**关键概念：**

| 概念 | 说明 | 类比 |
|------|------|------|
| `StateFlow<T>` | 只读的可观察数据流 | 只能看不能发的群公告 |
| `MutableStateFlow<T>` | 可写的可观察数据流 | 可以编辑的群公告 |
| `.value` | 获取/设置当前值 | 读/写群公告内容 |
| `.asStateFlow()` | 把可写的变成只读的 | 给群公告设"仅管理员可编辑" |

#### 5.4.2 在 GameViewModel 中添加 StateFlow

```kotlin
import kotlinx.coroutines.flow.MutableStateFlow

class GameViewModel : ViewModel() {
    // 这是一个可变的 StateFlow，只在 ViewModel 内部可以修改
    private val _uiState = MutableStateFlow(GameUiState())
}
```

**逐行解释：**

- `private val _uiState`：
  - `private` = 只能在 GameViewModel 类内部访问（外部看不见）
  - `_` 前缀 = Kotlin 的命名惯例，表示"这是私有的/内部的"
- `MutableStateFlow(GameUiState())`：
  - 创建一个"可变的"StateFlow
  - 初始值是 `GameUiState()` —— 所有字段使用默认值

### 5.5 步骤 5：后备属性（Backing Property）——"内外有别"

#### 5.5.1 什么是后备属性？

**后备属性 = 在类内部可以对数据进行修改，但对外部只能读取。**

一个简单的例子：

```kotlin
// 简单的后备属性示例
private var _count = 0          // 内部可变

val count: Int                  // 外部只读
    get() = _count
```

| 角度的 | 看到的 _count | 看到的 count |
|--------|--------------|-------------|
| ViewModel 内部 | `private var` 可变，可随意修改 | `val` 不可变，只能读取 |
| ViewModel 外部 | 看不见（private） | `val` 不可变，只能读取 |

**就像银行柜台：**
- 柜台里面（ViewModel 内部）：工作人员可以存取现金
- 柜台窗口（外部暴露的只读属性）：你只能看到余额，不能伸手进去拿钱

#### 5.5.2 为 uiState 添加后备属性

```kotlin
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow

class GameViewModel : ViewModel() {
    // 私有可变版本 —— 只有 ViewModel 内部能修改
    private val _uiState = MutableStateFlow(GameUiState())

    // 公开只读版本 —— 外部（UI）只能观察，不能修改
    val uiState: StateFlow<GameUiState> = _uiState.asStateFlow()
}
```

**逐行解释：**

- `val uiState: StateFlow<GameUiState>`：
  - 公开属性（没有 `private`，默认就是 public）
  - 类型是 `StateFlow<GameUiState>`，不是 `MutableStateFlow`——外部代码只能读不能写
- `= _uiState.asStateFlow()`：
  - `asStateFlow()` 将一个 `MutableStateFlow` 转换成只读的 `StateFlow`
  - 就像把一张可编辑的纸复印了一份，给外面的人看复印件（他们无法修改）

**对比总结：**

| 属性 | 可见性 | 类型 | 谁可以改 | 谁可以读 |
|------|--------|------|---------|---------|
| `_uiState` | private | MutableStateFlow | ViewModel | ViewModel |
| `uiState` | public | StateFlow | 没人（只读） | 所有人都可以 |

这就是**封装**的力量——外部代码无法不小心修改 ViewModel 的内部数据。

### 5.6 步骤 6：添加"选词"和"打乱"功能

现在 ViewModel 有了数据容器，但还没有任何游戏逻辑。接下来添加选词和打乱词的功能。

#### 5.6.1 添加内部变量

```kotlin
import com.example.android.unscramble.data.allWords

class GameViewModel : ViewModel() {
    // ... 之前添加的 StateFlow 代码 ...

    private lateinit var currentWord: String          // 当前正确词（用于比较）
    private var usedWords: MutableSet<String> = mutableSetOf()  // 已用过的词
}
```

**逐行解释：**

- `private lateinit var currentWord: String`：
  - `lateinit` = "late initialization"（延迟初始化）。告诉 Kotlin："我现在不赋值，但我保证在使用之前会赋值"
  - 为什么用 `lateinit`？因为我们不知道第一个词是什么，在 `init` 块中才从词库选取
  - `currentWord` 存储**正确的**原词（不是乱序后的），用于验证玩家的猜测
- `private var usedWords: MutableSet<String> = mutableSetOf()`：
  - `MutableSet` = 可修改版本的 Set（集合）
  - `mutableSetOf()` = 创建一个空的可变集合
  - 用途：记录这一局已经出现过的词，保证不重复

#### 5.6.2 选词函数：pickRandomWordAndShuffle()

```kotlin
private fun pickRandomWordAndShuffle(): String {
    // 第1步：从词库随机选一个词
    currentWord = allWords.random()

    // 第2步：检查这个词是否已经用过
    if (usedWords.contains(currentWord)) {
        // 用过了，重新选（递归调用自己）
        return pickRandomWordAndShuffle()
    } else {
        // 没用过——加入已用集合，打乱后返回
        usedWords.add(currentWord)
        return shuffleCurrentWord(currentWord)
    }
}
```

**逐行详细解释——像解释一本故事书：**

```
这就像一个抽奖游戏：
  你有一个装满单词卡片的大盒子（allWords）

  1. 闭着眼睛从盒子里抽一张（allWords.random()）
  2. 看看手上这张牌是不是之前已经抽过了（usedWords.contains()）
     - 如果抽过了 → 放回去，重新抽（递归调用自己）
     - 如果没抽过 → 记下来（add to usedWords），把这张牌打乱顺序（shuffle）
```

- `allWords.random()`：从 `Set` 中随机返回一个元素
- `if (usedWords.contains(currentWord))`：检查这个词是否在这一局中出现过
- `return pickRandomWordAndShuffle()`：**递归调用**——函数自己调用自己。这意味着"重来一次，直到选到没用过的词"
- `usedWords.add(currentWord)`：加入已用集合
- `return shuffleCurrentWord(currentWord)`：调用打乱函数（下面会定义）

> **什么是递归？** 递归就是函数自己调用自己，就像你照镜子——镜子里的你也在照镜子，无限循环。但在程序里，每次 `contains` 返回 false 时就停止递归了。

#### 5.6.3 打乱函数：shuffleCurrentWord()

```kotlin
private fun shuffleCurrentWord(word: String): String {
    // 第1步：把单词变成字符数组
    val tempWord = word.toCharArray()
    // 第2步：打乱数组
    tempWord.shuffle()
    // 第3步：检查打乱后是否和原词一样（比如原词 "aa" 打乱后还是 "aa"）
    while (String(tempWord).equals(word)) {
        tempWord.shuffle()   // 一样就重新打乱
    }
    // 第4步：返回打乱后的字符串
    return String(tempWord)
}
```

**逐行解释：**

```
原词："CRAZY"

1. toCharArray() → ['C', 'R', 'A', 'Z', 'Y']  （字符数组）
2. shuffle()     → ['Z', 'A', 'R', 'Y', 'C']  （随机打乱）
3. String(charArray) → "ZARYC"                 （重新拼成字符串）
```

- `word.toCharArray()`：字符串 → 字符数组。因为 `shuffle()` 是数组的方法
- `tempWord.shuffle()`：Kotlin 标准库函数，随机打乱数组元素顺序
- `while (String(tempWord).equals(word))`：
  - 把数组拼回字符串，检查是否和原词一样
  - 什么时候会一样？比如 `"aba"` 打乱后可能还是 `"aba"`
  - 如果一样，就继续打乱，直到不一样
- `return String(tempWord)`：把字符数组转换回字符串并返回

#### 5.6.4 重置游戏：resetGame()

```kotlin
fun resetGame() {
    usedWords.clear()          // 清空已用词集合
    _uiState.value = GameUiState(
        currentScrambledWord = pickRandomWordAndShuffle()
    )
}
```

**逐行解释：**

- `usedWords.clear()`：把 Set 清空——新一局重新开始
- `_uiState.value = GameUiState(...)`：
  - 直接给 `_uiState.value` 赋一个新值——这是**整体替换**整个状态
  - `pickRandomWordAndShuffle()` 被调用 → 选一个词、记录到 `usedWords`、打乱、返回乱序词
  - 新创建的 `GameUiState` 有一个乱序词，其他字段使用默认值（score=0, wordCount=1...）

#### 5.6.5 初始化块：init

```kotlin
init {
    resetGame()
}
```

**逐行解释：**

- `init { }`：Kotlin 的"初始化块"。在类的实例被创建时自动执行。
- `resetGame()`：一创建 ViewModel 就开始一局新游戏

**执行顺序：**
```
ViewModel 被创建（App 启动或配置更改后重建）
  → init 块执行
    → resetGame() 执行
      → pickRandomWordAndShuffle() 执行
        → 选词、打乱
      → _uiState.value 被赋值为新的 GameUiState
      → UI 观察到变化 → 界面显示乱序词
```

## 6. 构建 Compose 界面：单向数据流

### 6.1 什么是单向数据流（UDF — Unidirectional Data Flow）？

**UDF = 数据只往一个方向流动。像瀑布一样，水只能往下流。**

```
         ┌──────────────┐
         │  ViewModel   │  ← 数据在这里"生产"
         │  (上游)      │
         └──────┬───────┘
                │ 状态向下流 (State Down)
                ↓
         ┌──────────────┐
         │   UI 界面    │  ← 数据在这里"展示"
         │  (下游)      │
         └──────┬───────┘
                │ 事件向上冒 (Events Up)
                ↓
         ┌──────────────┐
         │  ViewModel   │  ← 收到事件 → 处理 → 更新状态
         │  (上游)      │     → 新状态向下流 → 循环
         └──────────────┘
```

**UDF 的三大步骤，一个循环：**

| 步骤 | 做什么 | 谁做 | 生活类比 |
|------|--------|------|---------|
| **事件 (Event)** | 用户点击按钮、输入文字 | UI 产生 | 你按了遥控器上的"音量+" |
| **更新状态 (Update State)** | 根据事件修改数据 | ViewModel 处理 | 电视内部把音量值 +1 |
| **显示状态 (Display State)** | 新数据显示到界面 | UI 更新 | 电视上显示音量条变了 |

**好处：**
- 数据流向清晰：出了问题你知道谁改了数据
- 没有循环依赖：UI 不直接改状态，ViewModel 不直接操作 UI
- 易于测试：ViewModel 可以脱离 UI 被单独测试

### 6.2 步骤 1：让 GameScreen 能访问 ViewModel

修改 `GameScreen.kt` 中的 `GameScreen()` 函数：

```kotlin
import androidx.lifecycle.viewmodel.compose.viewModel
import androidx.compose.runtime.collectAsState
import androidx.compose.runtime.getValue

@Composable
fun GameScreen(
    gameViewModel: GameViewModel = viewModel()  // ← 新增参数
) {
    // 收集 ViewModel 的状态
    val gameUiState by gameViewModel.uiState.collectAsState()

    // ... 其余代码暂时不动 ...
}
```

**逐行解释——每一个新 import 的含义：**

| import | 提供什么 | 为什么需要 |
|--------|---------|-----------|
| `viewModel` | `viewModel()` 函数 | 在 Compose 中创建/获取 ViewModel 实例 |
| `collectAsState` | `collectAsState()` 扩展函数 | 把 StateFlow 转成 Compose 能懂的 State |
| `getValue` | `by` 委托需要的操作符 | 让 `val x by ...` 语法能正常工作 |

**参数详解：**

```kotlin
gameViewModel: GameViewModel = viewModel()
```

- `gameViewModel: GameViewModel`：参数名是 `gameViewModel`，类型是 `GameViewModel`
- `= viewModel()`：**默认值**。如果调用 `GameScreen()` 时不传参数，就自动调用 `viewModel()` 创建一个

> **`viewModel()` 函数做了什么？**
> 1. 检查当前 Compose 上下文中是否已经有 GameViewModel 实例
> 2. 如果有 → 返回已有的（关键！这就是旋转屏幕后数据不丢的原因）
> 3. 如果没有 → 创建新的实例

**collectAsState 详解：**

```kotlin
val gameUiState by gameViewModel.uiState.collectAsState()
```

- `gameViewModel.uiState`：获取 ViewModel 的只读 StateFlow
- `.collectAsState()`：从 StateFlow 中"收集"最新值，返回一个 Compose 的 `State` 对象
- `by`：Kotlin 的"属性委托"语法。意思是"`gameUiState` 的值由 `collectAsState()` 返回的 State 对象来管理"
- 当 StateFlow 有新值发布时，Compose 会自动检测到变化，**重组**所有使用了 `gameUiState` 的可组合函数

> **重组 (Recomposition) 是什么？** Compose 的核心机制：当观察的数据变了，自动重新执行受影响的 `@Composable` 函数来更新界面。你不需要手动告诉界面"刷新"——它自己会刷。

这行代码有很多层含义，我们拆开看：

| 片段 | 含义 |
|------|------|
| `var` | 可变变量（与 `val` 相反，`val` 不能再赋值） |
| `userGuess` | 变量名——用户的猜测 |
| `by` | 属性委托关键字——"userGuess 的值由 mutableStateOf("") 来管理" |
| `mutableStateOf("")` | 创建一个 Compose 可观察的可变状态，初始值为 "" |
| `private set` | setter（赋值操作）是私有的——外部代码可以读但不能直接 `gameViewModel.userGuess = "hello"` |

> **为什么 `userGuess` 用 `mutableStateOf` 而不是放到 `GameUiState` 里用 `StateFlow`？**
>
> 这是一个设计选择：
> - `userGuess` 是**临时的 UI 交互状态**——只和输入框相关，不是"游戏的业务状态"
> - `GameUiState` 里的东西（分数、乱序词、单词数）是**游戏业务状态**
> - 分开管理让每个状态的更新不会影响另一个

`updateUserGuess()` 方法：
```kotlin

    userGuess = guessedWord
}
```
- 外部代码不能直接 `gameViewModel.userGuess = "hello"` （因为 `private set`）
- 但可以通过这个方法来间接设置
- 这保持了**封装性**——如果以后需要在设值时做额外处理（如日志），只改这一个方法就行

### 7.1 步骤 1-3：在 ViewModel 中添加验证逻辑

在 `GameViewModel.kt` 中添加 `checkUserGuess()` 方法：

```kotlin
import kotlinx.coroutines.flow.update

fun checkUserGuess() {
    // 比较用户输入和正确单词（忽略大小写）
    if (userGuess.equals(currentWord, ignoreCase = true)) {
        // 猜对了！——后面会在这里加加分逻辑
    } else {
        // 猜错了——更新状态，让界面显示错误提示
        _uiState.update { currentState ->
            currentState.copy(isGuessedWordWrong = true)
        }
    }
    // 无论对错，都清空输入框
    updateUserGuess("")
}
```

**逐行详细解释：**

```kotlin
if (userGuess.equals(currentWord, ignoreCase = true))
```

- `userGuess`：用户在输入框中打的内容
- `currentWord`：这一轮的正确单词（在 `pickRandomWordAndShuffle()` 中设置的）
- `.equals(...)`：比较两个字符串内容是否相同
- `ignoreCase = true`：忽略大小写。用户输入 "crazy" 和 "CRAZY" 都算对

```kotlin
_uiState.update { currentState ->
    currentState.copy(isGuessedWordWrong = true)
}
```

这是**不可变状态更新的标准模式**，我们一步步拆解：

| 代码 | 含义 |
|------|------|
| `_uiState.update { ... }` | StateFlow 的 update 方法。接受一个 lambda，在 lambda 中返回新的状态值。是原子操作——保证不会有多线程冲突 |
| `currentState ->` | lambda 参数——当前这一刻的状态快照 |
| `currentState.copy(...)` | `copy()` 方法创建一个新的 GameUiState 对象，**除指定字段外**其余字段和原状态相同 |
| `isGuessedWordWrong = true` | 只修改这一个字段 |
| `}` | lambda 返回的 `copy()` 结果成为新的状态值 |

**为什么不直接 `_uiState.value.isGuessedWordWrong = true`？**

因为 `GameUiState` 里的字段都是 `val`（不可变），编译就不允许直接修改！这是故意设计——强制你通过 `copy()` 创建新对象，保持不可变性。

#### 7.1.1 copy() 方法详解

```kotlin
// 一个简单的例子
data class User(val name: String, val age: Int)

val jack = User(name = "Jack", age = 1)
val olderJack = jack.copy(age = 2)

// olderJack.name == "Jack" (没变的保持原值)
// olderJack.age == 2       (只改了这一个)
```

**copy() 做了什么？**
1. 创建一个**新的** User 对象（原对象 jack 毫发无损）
2. 新对象中，`name` 被自动设置为 jack 的 name 值
3. `age` 被设置为参数中指定的新值

这就是"不可变数据"的魅力——你永远不会意外修改旧数据，因为每次修改都产生一个新的。

### 9.1 步骤 1-4：在 updateGameState 中添加游戏结束判断

```kotlin
import com.example.android.unscramble.data.MAX_NO_OF_WORDS   // = 10

private fun updateGameState(updatedScore: Int) {
    if (usedWords.size == MAX_NO_OF_WORDS) {
        // 10个词已经全部用完——游戏结束！

            currentState.copy(

        // 正常流程——还有词，选下一个

            currentState.copy(

                currentScrambledWord = pickRandomWordAndShuffle(),
                currentWordCount = currentState.currentWordCount.inc(),
                score = updatedScore
            )
        }
    }
}
```

**逐行解释：**

```kotlin
if (usedWords.size == MAX_NO_OF_WORDS)
```

- `usedWords`：一局中已经使用过的词集合
- `usedWords.size`：用过的词的数量。当 size == 10（MAX_NO_OF_WORDS）时，词库的 10 个名额已用完
- 注意：判断点在"选下一个词之前"——第 10 个词猜完（或跳过后），`usedWords.size` 就变成了 10

**游戏结束分支中 NOT 做的事：**
- **不选新词**：没有 `pickRandomWordAndShuffle()`
- **不增加单词数**：没有 `currentWordCount = ...inc()`
- 只做：保留分数 + 设置 `isGameOver = true`

```kotlin
    val currentWordCount: Int = 1,
    val score: Int = 0,
    val isGuessedWordWrong: Boolean = false,
```

### 9.3 显示游戏结束对话框

#### 9.3.1 FinalScoreDialog 详解（起始代码已提供）

```kotlin
@Composable
private fun FinalScoreDialog(
    score: Int,                          // 最终得分，显示在弹窗中
    onPlayAgain: () -> Unit,             // 点击"Play Again"时调用
    modifier: Modifier = Modifier
) {
    val activity = (LocalContext.current as Activity)  // 获取Activity引用，用于Exit

    AlertDialog(
        // 点击对话框外部或按返回键——什么都不做（玩家必须选择Exit或Play Again）
        onDismissRequest = { },

        // 标题行
        title = {
            Text(text = stringResource(R.string.congratulations))
        },

        // 正文——显示得分
        text = {
            Text(text = stringResource(R.string.you_scored, score))
        },

        modifier = modifier,

        // 左侧按钮——退出
        dismissButton = {
            TextButton(onClick = { activity.finish() }) {
                Text(text = stringResource(R.string.exit))
            }
        },

        // 右侧按钮——再来一局
        confirmButton = {
            TextButton(onClick = onPlayAgain) {
                Text(text = stringResource(R.string.play_again))
            }
        }
    )
}
```

**每个部分的详细解释：**

| 部分 | 代码 | 说明 |
|------|------|------|
| Activity 引用 | `val activity = (LocalContext.current as Activity)` | `LocalContext.current` 获取当前 Composable 的 Context。`as Activity` 类型转换。`activity.finish()` 关闭应用 |
| 标题 | `title = { Text(...) }` | 显示 "Congratulations!" |
| 正文 | `text = { Text(stringResource(R.string.you_scored, score)) }` | 显示 "You scored: 160"——分数是动态的 |
| 退出按钮 | `dismissButton = { TextButton(onClick = { activity.finish() }) { ... } }` | 调用 Activity 的 `finish()` 方法关闭应用 |
| 重玩按钮 | `confirmButton = { TextButton(onClick = onPlayAgain) { ... } }` | 调用传入的 `onPlayAgain` 回调 |

> **为什么 title、text、按钮都是 `{ ... }` 的 lambda 形式而不是直接传值？**
> 这是 Compose 的"槽位（slot）API"模式。使用 lambda 可以让内容在需要时才构建，且可以自动响应状态变化。


- 当 `isGameOver` 变回 `false` 时，对话框自动消失


- 用户点击 "Play Again" 时，调用 ViewModel 的 `resetGame()` 方法
- `resetGame()` 清空 `usedWords`，重新选词，创建新的初始状态

### 9.4 完善 resetGame()

```kotlin
fun resetGame() {
    usedWords.clear()                    // 清空已用词集合
    // 整体替换状态为全新的初始状态
    _uiState.value = GameUiState(
        currentScrambledWord = pickRandomWordAndShuffle()
    )
}
```

**为什么这里用 `_uiState.value = ...` 而不是 `_uiState.update { ... }`？**

两种写法都可以，但这里使用直接赋值更合适：
- `_uiState.value = GameUiState(...)`：创建全新的初始状态对象
- `_uiState.update { it.copy(...) }`：基于旧状态修改部分字段

重置游戏就是"创建一个全新的初始状态"，不需要基于旧状态复制，所以直接用 `=` 赋值。

**resetGame 的效果：**

```
重置前：                                        重置后：
{                                                {
  currentScrambledWord: "LOOBALN"                 currentScrambledWord: "HCNEB" ← 新词
  currentWordCount: 10                            currentWordCount: 1           ← 回到第1轮
  score: 160                                      score: 0                      ← 归零
  isGuessedWordWrong: false                       isGuessedWordWrong: false
  isGameOver: true                                isGameOver: false             ← 游戏重新开始
}                                                }
```

## 10. 设备旋转与状态保留：ViewModel 的超能力

### 10.1 为什么旋转后数据不丢？

这是 ViewModel 最核心的超能力。让我们深入理解。

**没有 ViewModel 时：**

```
竖屏游戏 → 旋转 → Activity 销毁
                      ↓
                  新的 Activity 创建
                      ↓
                  从头开始 → 分数归零、词回到第1个
```

**有 ViewModel 时：**

```
竖屏游戏 → 旋转 → Activity 销毁 ✗
                   ViewModel 继续存活 ✓
                      ↓
                  新的 Activity 创建
                      ↓
                  viewModel() 返回已有的 ViewModel（不是创建新的！）
                      ↓
                  collectAsState() 读到之前的状态数据
                      ↓
                  界面恢复 → 分数不变、词不变
```

**关键机制：`viewModel()` 函数**

```kotlin
gameViewModel: GameViewModel = viewModel()
```

`viewModel()` 不是每次都创建新实例：
1. 第一次调用 → 创建新的 ViewModel → 存储到当前 Compose 上下文中
2. Activity 重建后调用 → 发现上下文中已有 ViewModel → **返回已有的**

就像：
```
你去图书馆存包：
  第一次 → 拿到一个储物柜钥匙 → 把包放进去
  中途出去上了个厕所回来 → 用同一把钥匙 → 打开同一个柜子 → 包还在
```

### 10.2 验证

1. 运行应用
2. 玩几轮——猜对几个词，拿到一些分数
3. 旋转设备（或用模拟器的旋转按钮）
4. 观察：分数、单词数、当前乱序词——**全部保持不变！**

### 10.3 状态保留方案对比

| 方案 | 保留时机 | 能存什么 | 使用场景 |
|------|---------|---------|---------|
| `remember` | 只在同一次**重组**期间 | 简单值 | 临时的 UI 局部状态（如折叠/展开） |
| `rememberSaveable` | **重组** + **配置更改** | 能被 Bundle 保存的类型（基本类型、String 等） | 需要跨旋转保存的简单 UI 状态 |
| **ViewModel + StateFlow** | **独立于 Activity 生命周期** | 任意复杂的数据结构 | **应用业务逻辑和数据** ← 推荐！ |

> **什么时候用哪个？**
> - 一个开关的"开/关"状态 → `remember` 就够了
> - 输入框的文字（旋转后保留） → `rememberSaveable`
> - 游戏分数、进度、数据 → `ViewModel`

### 12.1 核心概念速查表

| 概念 | 用一句话解释 | 生活类比 |
|------|-------------|---------|
| **分离关注点** | 不同的事给不同的类做 | 厨房里主厨、切菜工、服务员各司其职 |
| **通过模型驱动界面** | 数据决定界面长什么样 | 遥控器控制电视内部的频道，电视画面跟着变 |
| **ViewModel** | 存储界面状态的"数据管家" | 银行保险柜——柜台关了重建，保险柜还在 |
| **UI State** | 用不可变数据描述界面 | 给界面拍一张"数据X光片" |
| **StateFlow** | 会主动通知观察者的数据容器 | 群聊公告——改了内容所有人"叮"一声收到 |
| **后备属性** | 内部可变、外部只读 | 银行柜台——你只能看余额，不能伸手拿钱 |
| **UDF** | 状态向下流，事件向上冒 | 瀑布只能往下流，水蒸气只能往上升 |
| **不可变性** | 创建后不能改，改就新建副本 | 拍完的照片不能涂改，要修图就另存一张新的 |
| **copy()** | 基于旧对象创建只改部分字段的新对象 | 复印一张文件，改一个数字，原件不变 |
| **mutableStateOf** | Compose 的可观察可变状态 | 贴了传感器的变量——值一变，界面就刷新 |
| **collectAsState()** | 把 StateFlow 转为 Compose 状态 | 翻译官——把"群聊语言"翻译成"Compose 语言" |

### 12.3 完整的数据流回顾

让我们看一次"用户猜对词"的完整数据流：

```
1. 用户输入 "crazy"  →  onValueChange  →  updateUserGuess("crazy")
                                                    ↓
2. 用户点 Submit     →  onClick         →  checkUserGuess()
                                                    ↓
3. "crazy" == "CRAZY"? →  忽略大小写比较  →  YES! 猜对了
                                                    ↓
4. 计算新分数: 当前分 + 20                           ↓
                                                    ↓
5. updateGameState(新分数)
   ├── usedWords.size == 10? → 否，还有词
   └── copy(
          isGuessedWordWrong = false,
          currentScrambledWord = 新选词并打乱,
          currentWordCount = 原计数 + 1,
          score = 新分数
       )
                                                    ↓
6. updateUserGuess("")  →  清空输入框               ↓
                                                    ↓
7. _uiState 发出新状态                               ↓
                                                    ↓
8. collectAsState() 检测到变化                       ↓
                                                    ↓
9. gameUiState 更新  →  GameLayout 重组              ↓
   - currentScrambledWord → 新词显示
   - wordCount → 更新为 +1
   - score → GameStatus 更新显示
   - isGuessedWordWrong = false → 输入框恢复正常颜色
                                                    ↓
10. 界面显示新状态，等待下一轮用户输入 ← 循环回到步骤1
```
