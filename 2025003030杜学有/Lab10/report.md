# Lab10 为 Lunch Tray 添加导航 实验报告


## 一、实验概述
本次实验基于Jetpack Compose Navigation导航组件，在已有Lunch Tray点餐UI与订单ViewModel基础上，完成五页面完整导航开发。应用包含开始点餐、选择主菜、选择配菜、选择佐餐、订单结账五大页面，实现页面跳转、顶部导航栏动态展示、返回堆栈管理、订单状态联动重置等功能，熟悉Compose多页面应用标准开发流程。

## 二、核心原理与概念
### 2.1 NavController、NavHost、composable三者关系
1. **NavController**：导航核心控制器，通过`rememberNavController()`创建，统一管控页面跳转、页面返回、返回栈管理，是所有导航行为的调度中心。
2. **NavHost**：导航承载容器，依附布局内部，绑定导航控制器，指定应用默认启动页面，负责渲染对应路由页面内容。
3. **composable**：路由注册函数，为每一个业务页面绑定唯一路由标识，声明页面跳转入口与回调逻辑。

三者协作流程：NavController发起跳转指令 → NavHost匹配路由 → 加载composable注册对应的页面组件，完成页面切换。

### 2.2 导航枚举类设计思路
本次实验采用`enum class LunchTrayScreen`统一管理全部页面，为每个页面绑定字符串资源标题。
选用枚举而非原生字符串路由原因：
1. 具备**类型安全**，编译阶段即可排查路由拼写错误；
2. 页面路由、页面标题集中统一管理，代码结构清晰，便于后期维护；
3. 可直接通过枚举实例快速获取页面名称与标题资源，适配顶部标题动态切换；
4. 新增页面仅需追加枚举项，拓展性更强。

## 三、功能组件设计实现
### 3.1 自定义顶部导航栏 LunchTrayAppBar
1. 动态标题：根据当前所在页面枚举值，自动读取字符串资源展示页面名称；
2. 返回按钮逻辑：通过判断`previousBackStackEntry`是否为空，**首页隐藏返回键**，其余页面正常展示返回图标；
3. 点击返回调用`navigateUp()`，遵循安卓原生返回交互逻辑，同时适配Material3主题样式。

### 3.2 整体导航业务流程
1. 基础流转流程
`开始页面(Start) → 选择主菜(Entree) → 选择配菜(SideDish) → 选择佐餐(Accompaniment) → 结账页面(Checkout)`
2. 按钮交互规则
- Start页面点击Start Order：跳转至主菜选择页，并清理首页栈记录；
- 菜品选择页点击Next：进入下一点餐流程页面；
- 任意页面点击Cancel：直接返回首页，同时清空已选订单数据；
- 结账页面点击提交订单：返回首页并重置订单，完成一次点餐流程。

### 3.3 返回堆栈优化策略
1. 从首页进入点餐流程时，使用`popUpTo`将首页弹出返回栈，用户按下系统返回键直接退出应用，避免重复退回首页；
2. 取消订单、提交订单统一清空点餐流程页面栈，直接回到应用初始首页，保证点餐流程独立互不干扰；
3. 页面跳转同步联动OrderViewModel，页面回退同步重置订单价格、菜品选择数据。

## 四、核心代码实现说明
1. 定义导航枚举类，绑定五个页面对应标题资源；
2. 初始化导航控制器，监听导航栈状态，实时获取当前所在页面；
3. 在Scaffold脚手架中嵌入自定义顶部导航栏，完成页面基础布局；
4. 通过NavHost配置全部页面路由，绑定页面点击回调；
5. 页面菜品选择回调绑定ViewModel更新方法，实现订单价格实时计算；
6. 取消、提交订单逻辑中完成页面跳转与订单数据双重重置。

## 五、实验遇到的问题及解决方法
### 问题1：从首页进入点餐页后，按返回键仍退回首页
**解决**：在首页跳转主菜页面的导航代码中添加`popUpTo`配置，将首页从返回堆栈中移除，实现返回直接退出应用。

### 问题2：点击Cancel返回首页，已选菜品数据依旧保留
**解决**：在取消订单跳转逻辑中，同步调用`viewModel.resetOrder()`方法，清空订单所有选中项与价格数据。

### 问题3：顶部导航栏页面标题无法自动刷新
**解决**：使用`currentBackStackEntryAsState()`实时监听导航栈变化，动态匹配当前页面枚举，实现标题实时更新。

### 问题4：页面多级跳转后，返回路径混乱错乱
**解决**：统一规范所有退出流程路由写法，强制清空点餐流程页面栈，固定跳转目标为首页，统一导航逻辑。
